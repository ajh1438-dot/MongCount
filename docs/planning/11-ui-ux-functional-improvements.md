# 11. UI/UX 및 기능 개선사항 명세

> 프로젝트: MongCount (쉼일지)
> 작성일: 2026-04-18
> 기준: 실제 구현 상태 점검 + `docs/planning/06-screens.md` + `specs/domain/resources.yaml`
> 목적: 현재 앱 전반을 살펴본 뒤, 제품 완성도를 높이기 위한 UI/UX 및 기능 개선 항목을 우선순위와 완료 기준까지 포함해 정의한다.

---

## 1. 요약

현재 MongCount는 다음 범위가 이미 의미 있게 연결되어 있다.

- S1 홈 → S2 쉼 → S3 리뷰 저장의 핵심 흐름
- S5 아카이브 최근 7일 조회 + free/premium 경계
- S6 주간 리포트 기본 UI + 월간 탭 게이팅
- S9 설정의 테마/알림 슬롯 영속화 + 최소 로그아웃
- S10 Paywall 기본 UI + source 기반 복귀
- S11 빈상태/네트워크/알림 거부 배너 일부 적용

반면, 아직 제품 완성도를 떨어뜨리는 큰 공백도 분명하다.

- 일부 화면이 실제 데이터 기반이 아니라 고정 문구/플레이스홀더에 머무름
- 결제/계정/정책/법적 영역이 아직 신뢰 가능한 실제 제품 수준이 아님
- 몇몇 상호작용이 "동작은 하지만 차분하고 완성된 경험"까지는 아님
- 화면별 상태 설계는 있지만, 로딩/빈결과/실패/복귀 경험의 일관성이 부족함

이 문서는 위 상태를 바탕으로 개선 항목을 **P0 즉시 보완**, **P1 제품 완성도 향상**, **P2 확장 가치**로 나눈다.

## 3. 우선순위 분류

### P0 — 출시 전 반드시 보완할 항목
사용자가 실제 제품으로 느끼기 위해 반드시 닫아야 하는 항목

### P1 — 제품 완성도를 크게 높이는 항목
핵심 플로우는 돌아가지만, 유지율/신뢰/몰입감을 높이기 위해 필요한 항목

### P2 — 확장 가치 및 프리미엄 강화 항목
기본 경험 이후 확장 가치, 유료 전환, 장기 유지율에 기여하는 항목

## 4. 개선사항 상세 명세

## A. 핵심 플로우 (S1 → S2 → S3)

### A-1. 리뷰 화면의 세션 맥락 정확도 개선
- 우선순위: P0
- 현재 상태:
  - `components/screens/rest-review/rest-review-screen.tsx`의 헤드라인이 항상 `5분을 마치셨어요 ✓`로 고정되어 있다.
  - 실제로는 3분/5분/10분 세션이 모두 존재한다.
- 문제:
  - 사용자가 방금 수행한 실제 쉼 시간과 리뷰 화면 카피가 어긋난다.
  - 작은 불일치지만 제품 신뢰감을 떨어뜨린다.
- 개선 요구:
  - 헤드라인과 보조 문구를 실제 `durationMinutes` 값 기준으로 렌더링한다.
  - 중단 진입(`fromAbort`)일 때도 문구가 자연스럽게 바뀌어야 한다.
- 완료 기준:
  - 3/5/10분 각각에서 리뷰 헤드라인이 맞게 노출된다.
  - 중단 진입 시에도 시간/상태 문구가 서로 모순되지 않는다.

### A-2. 타이머 중단 경험을 브라우저 기본 confirm에서 제품 UI로 교체
- 우선순위: P0
- 현재 상태:
  - `components/screens/s2/s2-rest-timer-screen.tsx`에서 중단 확인이 `window.confirm()`에 의존한다.
- 문제:
  - 브라우저 기본 confirm은 모바일 PWA 분위기를 깨고, 시각/톤 일관성이 없다.
  - 차분한 제품 톤과 맞지 않는다.
- 개선 요구:
  - 하단 시트 또는 인라인 확인 카드로 전환한다.
  - 선택지는 최소 `계속 쉬기` / `중단하고 기록하기` 2개로 유지한다.
- 완료 기준:
  - 브라우저 기본 confirm이 제거된다.
  - 모바일에서 한 손으로 닫기 쉬운 UI로 바뀐다.
  - 실수 중단률을 낮추는 문구가 포함된다.

### A-3. 저장 후 복귀 경험의 명확화
- 우선순위: P1
- 현재 상태:
  - 저장 후 2초 토스트 뒤 홈으로 자동 이동한다.
- 문제:
  - 자동 이동 자체는 좋지만, 사용자가 저장 성공을 짧은 순간 놓치면 흐름이 갑자기 바뀐 느낌을 받을 수 있다.
- 개선 요구:
  - 저장 직후 피드백을 더 명확하게 하고, 홈 복귀 이유를 보여준다.
  - 예: `저장했어요. 홈으로 돌아갈게요.` 문구를 더 안정적으로 유지.
- 완료 기준:
  - 저장/건너뛰기 후 복귀가 놀랍지 않다.
  - 토스트/버튼 비활성 상태/자동 이동 타이밍이 서로 충돌하지 않는다.

### A-4. 홈 배너 우선순위와 중첩 규칙 정리
- 우선순위: P1
- 현재 상태:
  - `components/screens/s1/s1-home-screen.tsx`에서 게스트 배너와 알림 거부 배너가 함께 나타날 수 있다.
- 문제:
  - 작은 화면에서 상단 정보가 과밀해질 수 있다.
- 개선 요구:
  - 배너 우선순위를 정의한다.
  - 예: 알림 거부 > 게스트, 또는 두 배너 동시 노출 시 1개는 축약형으로 처리.
- 완료 기준:
  - 홈 상단이 2개 이상 배너로 무겁게 보이지 않는다.
  - 중요한 안내가 묻히지 않는다.

## B. 아카이브 / 리포트 경험

### B-1. 아카이브의 "검색 결과 없음" 상태 추가
- 우선순위: P0
- 현재 상태:
  - `app/(tabs)/archive/page.tsx`는 필터/검색 후 결과가 0건이면 일반 빈 아카이브 상태로 보인다.
- 문제:
  - 실제로는 기록이 있는데, 검색 결과만 없는 경우와 완전 신규 유저 상태가 구분되지 않는다.
- 개선 요구:
  - `기록은 있지만 현재 조건에 맞는 결과가 없어요.` 상태를 별도로 제공한다.
  - `필터 초기화` 액션도 함께 제공한다.
- 완료 기준:
  - 완전 빈 상태 / 검색 결과 없음 / 오프라인 상태가 명확히 구분된다.

### B-2. 아카이브 필터 UX 명확화
- 우선순위: P1
- 현재 상태:
  - 상단에 `필터` 버튼이 있으나 실제 기능은 빠른 칩 위주이며, 버튼 자체는 실질 행동이 없다.
- 문제:
  - 사용자가 필터 버튼을 기대하고 눌렀을 때 비어 있는 경험을 한다.
- 개선 요구:
  - `필터` 버튼을 제거하거나, 실제 필터 패널을 열도록 만든다.
  - 빠른 칩과 고급 필터의 역할을 분리한다.
- 완료 기준:
  - 필터 UI의 진입점이 하나의 명확한 모델로 정리된다.

### B-3. 아카이브 카드 정보 밀도 재조정
- 우선순위: P1
- 현재 상태:
  - 카드에 시간 / 맑기 / 메모 / 타이머 길이가 함께 들어간다.
- 문제:
  - 작은 화면에서 텍스트 길이가 길어지면 카드 리듬이 약간 무거워진다.
- 개선 요구:
  - 1차 정보(시간, 메모)와 2차 정보(맑기, 타이머)를 시각적으로 더 분리한다.
  - 맑기 표현은 현재 `☀︎` 기반을 유지하되 가독성을 높인다.
- 완료 기준:
  - 카드 스캔 속도가 빨라진다.
  - 5~10개 카드가 연속으로 보여도 피로감이 적다.

### B-4. 주간 리포트의 실제 데이터 연결 강화
- 우선순위: P0
- 현재 상태:
  - `components/screens/s6/s6-weekly-report-screen.tsx`는 주간 범위, 도트, 평균 맑기, AI 코멘트가 고정값 위주다.
- 문제:
  - 리포트가 진짜 사용자 기록을 반영한다는 신뢰가 약하다.
- 개선 요구:
  - 최소한 날짜 범위, completed_days, avg_clarity, best_session은 실제 집계 기반으로 연결한다.
  - 고정 카피는 placeholder가 아닌 데이터 기반 fallback으로 전환한다.
- 완료 기준:
  - 기록 변화에 따라 주간 리포트 내용이 실제로 바뀐다.

### B-5. 월간 리포트 플레이스홀더 탈피
- 우선순위: P1
- 현재 상태:
  - premium + monthlyTabActive일 때도 `자리만 연결`된 상태다.
- 문제:
  - 유료 가치 제안이 약하다.
- 개선 요구:
  - 최소 버전이라도 4주 비교, 월간 평균 맑기, 총 세션 수, 최고/최저 기록을 제공한다.
- 완료 기준:
  - 프리미엄 사용자가 월간 탭에서 실제 가치 있는 내용을 본다.

## C. 설정 / 계정 / 신뢰 영역

### C-1. 설정 화면에 즉시 피드백 추가
- 우선순위: P0
- 현재 상태:
  - 알림 슬롯 토글은 영속화되지만 저장되었는지 시각 피드백이 거의 없다.
- 문제:
  - 사용자는 토글이 저장되었는지 확신하기 어렵다.
- 개선 요구:
  - 저장 성공을 짧은 토스트 또는 `저장됨` 마이크로카피로 보여준다.
  - 실패 시 복구 문구도 필요하다.
- 완료 기준:
  - 토글/테마 변경 후 저장 여부를 사용자가 1초 내 이해할 수 있다.

### C-2. 로그아웃 완료 피드백 표시
- 우선순위: P1
- 현재 상태:
  - `/auth/logout`은 `/settings?auth=signed_out`로 돌아오지만, 현재 `app/(tabs)/settings/page.tsx`에서 해당 쿼리 피드백을 보여주지 않는다.
- 문제:
  - 로그아웃 후 상태 변화가 시각적으로 불충분하다.
- 개선 요구:
  - `로그아웃되었어요.` 수준의 차분한 피드백을 상단 배너 또는 토스트로 노출한다.
- 완료 기준:
  - 로그아웃 직후 사용자가 액션 성공을 명확히 인식한다.

### C-3. 계정 삭제는 UI 비활성/보류 상태를 명확히 표기
- 우선순위: P0
- 현재 상태:
  - 설정에 `계정 삭제` 버튼이 있으나 실제 동작은 없다.
- 문제:
  - 실제 제품에서 가장 위험한 "동작하지 않는 위험 버튼" 상태다.
- 개선 요구:
  - 백엔드 지원 전까지는 버튼을 숨기거나 `준비 중` 상태로 바꾸고 설명을 붙인다.
- 완료 기준:
  - 오작동하거나 허위 기대를 만드는 위험 액션이 남아있지 않다.

### C-4. 정책/약관 링크 실제화
- 우선순위: P0
- 현재 상태:
  - 설정 하단 정책 링크가 `example.com`이다.
- 문제:
  - 신뢰, 법적 완성도, 결제 전환 모두에 악영향.
- 개선 요구:
  - 실제 정책/약관 문서를 연결하거나, 없으면 임시 페이지라도 앱 내부에 생성한다.
- 완료 기준:
  - 유저가 눌렀을 때 실제 문서를 볼 수 있다.

## D. 온보딩 / 로그인 / 전환율

### D-1. 온보딩 카피 오탈자/문장 다듬기
- 우선순위: P1
- 현재 상태:
  - 일부 카피에 자연스럽지 않은 표현과 오탈자 가능성이 있다. 예: `오후 경항...`, `iPhone 사용자인가요?` 등.
- 문제:
  - 브랜드 톤의 정제도가 떨어진다.
- 개선 요구:
  - 전체 카피를 차분하고 부드러운 톤으로 재정리한다.
  - 과장된 연구 인용/어색한 표현도 정돈한다.
- 완료 기준:
  - 전 화면의 카피 톤이 일관되고 자연스럽다.

### D-2. 로그인 화면의 선택 이유 보강
- 우선순위: P1
- 현재 상태:
  - `components/screens/s8/s8-login-screen.tsx`는 기능 설명은 있으나, 로그인/비로그인 선택의 차이가 충분히 설명되지는 않는다.
- 문제:
  - 사용자가 왜 로그인해야 하는지, 나중에 해도 되는지 판단 근거가 약하다.
- 개선 요구:
  - `로그인 시: 여러 기기 동기화`, `비로그인 시: 이 기기에서만 기록` 수준의 비교 설명을 추가한다.
- 완료 기준:
  - 로그인 전환을 강요하지 않으면서도 선택 이유가 명확하다.

### D-3. 온보딩 완료 후 첫 행동 유도 강화
- 우선순위: P1
- 현재 상태:
  - 온보딩 후 로그인 화면으로 이동하지만, 첫 쉼 시작까지의 감정적 연결이 약간 끊긴다.
- 문제:
  - 학습 흐름과 실제 행동 시작 사이에 온도 차가 생긴다.
- 개선 요구:
  - 로그인 화면 또는 홈 첫 진입에서 `지금 3분 쉬기`와의 연결감을 높인다.
- 완료 기준:
  - 온보딩 → 로그인/스킵 → 첫 쉼 시작의 흐름이 더 자연스럽다.

## E. Paywall / 유료 전환

### E-1. CTA 실동작 연결 또는 명확한 준비중 상태 제공
- 우선순위: P0
- 현재 상태:
  - `components/screens/s10/s10-paywall-screen.tsx`의 `7일 무료 체험 시작` 버튼은 실질 결제 흐름에 연결되어 있지 않다.
- 문제:
  - 결제 직전 화면에서 핵심 CTA가 비어 있으면 신뢰가 크게 떨어진다.
- 개선 요구:
  - 실제 checkout 연결 전이라면 `준비 중` 상태/대기 등록/문의 동선으로 바꾼다.
  - 실제 연결 시에는 trial/price/legal copy를 재검증한다.
- 완료 기준:
  - 유료 CTA가 실제 액션 또는 명확한 안내 중 하나를 제공한다.

### E-2. Paywall source별 맞춤 문구 제공
- 우선순위: P1
- 현재 상태:
  - source 기반 복귀는 되지만, 진입 이유에 따라 헤드라인/보조 카피는 동일하다.
- 문제:
  - 아카이브 때문에 들어온 사용자와 월간 리포트 때문에 들어온 사용자의 맥락이 다르다.
- 개선 요구:
  - `archive_boundary`, `weekly_monthly_tab`, `settings`별 서브카피를 분기한다.
- 완료 기준:
  - 진입 이유와 가치 제안이 자연스럽게 연결된다.

### E-3. 프리미엄 가치 설명의 실제 경험 중심 재작성
- 우선순위: P1
- 현재 상태:
  - 기능 목록은 있으나 사용자가 얻게 될 변화가 덜 보인다.
- 문제:
  - 기능 나열형 paywall은 전환력이 낮다.
- 개선 요구:
  - `기록을 7일 이상 보관`, `월간 패턴을 한 번에 보기`, `기기 바꿔도 이어보기`처럼 결과 중심 문장으로 재작성한다.
- 완료 기준:
  - paywall을 보고 premium의 효익이 5초 안에 이해된다.

## F. 공통 UX / 정보 구조 / 접근성

### F-1. 로딩/비동기 상태의 공통 패턴 정의
- 우선순위: P0
- 현재 상태:
  - 일부 화면은 바로 비거나 바로 렌더되며, 로딩 상태가 명시적이지 않다.
- 문제:
  - 느린 네트워크나 초기 hydration에서 화면이 불안정하게 느껴질 수 있다.
- 개선 요구:
  - 최소 skeleton/placeholder 패턴을 공통 정의한다.
- 완료 기준:
  - S1, S5, S6, S9, S10 진입 시 로딩 인지가 자연스럽다.

### F-2. 제품 내 문구/톤 일관성 가이드 정리
- 우선순위: P1
- 현재 상태:
  - 차분한 문구와 기능 설명 문구가 섞여 있고, 일부 표현은 아직 기획 문서 톤에 가깝다.
- 문제:
  - MongCount의 "조용한 회복" 브랜드 감각이 화면마다 균일하지 않다.
- 개선 요구:
  - 마이크로카피 원칙을 별도 가이드로 정리한다.
  - 금지 톤: 압박, 성취 강요, 과한 생산성 문구
  - 권장 톤: 회복, 관찰, 여유, 비판단적 안내
- 완료 기준:
  - 핵심 화면 카피가 동일한 브랜드 목소리를 갖는다.

### F-3. 접근성 세부 보강
- 우선순위: P1
- 현재 상태:
  - 기본 ARIA는 많이 들어가 있으나, 상태 메시지/토글/슬라이더/읽기 전용 재생의 맥락 설명은 더 다듬을 여지가 있다.
- 개선 요구:
  - 토글 변경 후 상태 읽기, readonly 모드 설명, 저장 완료 메시지, 배너 액션 포커스 이동을 점검한다.
- 완료 기준:
  - 스크린리더 기준 핵심 플로우의 의미 손실이 없다.

### F-4. 태블릿/데스크톱 보정
- 우선순위: P2
- 현재 상태:
  - 모바일 중심 max-width 구조는 좋지만, 태블릿 이상에서 여백/세로 밀도가 다소 허전할 수 있다.
- 개선 요구:
  - 카드 폭, 여백, 섹션 간격을 태블릿 기준으로 재조정한다.
- 완료 기준:
  - 모바일 우선 철학은 유지하면서도 큰 화면에서 어색하지 않다.

## 5. 기능 개선 명세

## G. 아직 비어 있는 핵심 기능

### G-1. 실제 결제 플로우 구현
- 우선순위: P0
- 범위:
  - checkout 시작
  - trial 상태 반영
  - 결제 성공/실패/취소 처리
  - 구독 상태 변경 반영
- 완료 기준:
  - S10 CTA부터 `subscriptions` 상태까지 실제로 이어진다.

### G-2. 계정 삭제 보안 플로우 구현
- 우선순위: P0
- 범위:
  - 최근 재인증
  - 이중 확인
  - 사용자 데이터 삭제 또는 비식별화 정책
  - 세션 종료
- 완료 기준:
  - 보안/법무/데이터 정합성이 보장되는 삭제 흐름 완성

### G-3. 주간/월간 리포트 실제 집계 연결
- 우선순위: P0
- 범위:
  - 실제 rest_sessions 기반 집계
  - premium gating과 연결
  - best session / avg clarity / totals 반영
- 완료 기준:
  - 리포트가 실제 사용 기록과 함께 살아 움직인다.

### G-4. 알림 설정과 실제 푸시 상태 연결
- 우선순위: P1
- 현재 상태:
  - 설정 슬롯 토글은 저장되지만, 실제 푸시 구독/스케줄 상태까지는 사용자에게 충분히 보이지 않는다.
- 개선 요구:
  - 현재 알림 권한 상태, 기기 구독 상태, 슬롯별 실제 발송 가능 여부를 보여준다.
- 완료 기준:
  - 사용자가 "토글은 켰는데 왜 알림이 안 오지?"를 스스로 이해할 수 있다.

### G-5. readonly replay 경험 고도화
- 우선순위: P1
- 현재 상태:
  - S5에서 S2 readonly 진입은 되지만, 재생 경험이 매우 간단하다.
- 개선 요구:
  - 해당 세션의 메모/맑기/완료 여부를 함께 보여주거나, 재생과 원기록의 연결을 더 분명히 한다.
- 완료 기준:
  - 과거 쉼을 "다시 보는" 가치가 명확해진다.

## 8. 결론

MongCount는 이미 핵심 구조와 제품 감각의 방향은 잘 잡혀 있다. 다만 현재 상태는 **좋은 프로토타입과 초도 제품 사이**에 있다. 다음 단계는 기능을 더 많이 붙이는 것이 아니라,

- 실제 데이터 기반으로 보이게 만들고
- 위험한 빈 버튼/플레이스홀더를 제거하고
- 작지만 신뢰를 주는 피드백을 채우고
- 결제/계정/정책처럼 제품 신뢰를 좌우하는 영역을 마무리하는 것

에 집중해야 한다.

이 명세는 그 우선순위를 기준으로 정리한 개선 백로그다.
